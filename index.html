The proposal

Put an AI agent in front of the call-centre queue

What we're introducing to the bank, in one screen — then the evidence that it works (below).

The problem you already have

Most inbound calls are routine — balances, card blocks, "was this me?", branch hours — yet every one of them queues for a consultant. Volume spikes (payday, fraud scares, outages) blow out hold times exactly when customers are most anxious, and hiring to the peak means idle capacity the rest of the month.

What we're introducing

An AI voice agent that answers the bank's number instantly, at any concurrency. It verifies the caller before a word of account information, resolves routine requests end-to-end through the bank's own systems — check a balance, block a lost card, file a dispute, answer product questions — and hands anything complex, sensitive, or judgement-shaped to a consultant with a summary, so nobody repeats themselves.

It deploys alongside the existing call centre and absorbs the routine tier and the overflow. It moves no money, gives no advice, and never guesses: what it can't ground in a system call, it escalates.

What the bank gets

Zero hold time on the routine tier; consultants freed for the complex, revenue-relevant conversations; a supervisor dashboard running on the same KPIs as the human channel (containment, handle time, escalations); and a PII-masked audit trail for every call. The ask: a scoped pilot on one call type — card blocks is ideal — with success criteria agreed up front and read off that same dashboard.

The section the risk team reads first

Security by construction, not by prompt

Every guarantee is graded: structural (the capability does not exist) beats process (it's checked and logged) beats contractual (it's in the DPA). "Please don't" is not on the list — and each guarantee below is demonstrable in the live demo today.

Structural Identity before information — in tiers

Before verification the agent has no account tools at all — a prompt injection cannot call a tool that doesn't exist. Three failed attempts triggers a security lockout event and the only remaining path is a human. Verification unlocks reads only: any account action (card block, dispute) requires step-up — a one-time code to the registered banking app, live in the demo today. A possession factor, not another passphrase: account number + ID digits + a passphrase are all "something you know," and two knowledge factors are not MFA (NIST 800-63B). App delivery, not SMS — SIM-swap is South Africa's dominant OTP attack. Gate placement is a policy switch, demoable both ways: actions-only (risk-proportionate, the PSD2/FFIEC pattern) or before any account data; production makes it risk-based — the gate escalates when fraud signals fire (recent SIM swap, prior fraud marker).

Structural No money movement — no such tool

There is no transfer or payment tool on any agent. Capability expansion is a governance decision the bank takes per-tool with step-up authentication — never a model behaviour.

Structural PII never leaves unmasked

Known values (the account number a caller just said) are masked exactly; regex nets catch SA-ID/Omang/account shapes; and a spoken-number net masks digits read out as words ("one double zero two three…") — the form voice transcripts actually contain — so even a supervisor reading the transcript never sees the caller's identifiers. The agent is also instructed never to echo them back. Precedent at scale: Wells Fargo runs 245M interactions/year with zero PII sent to the LLM — the same pattern.

Structural Grounded answers only

The agent may only state account facts returned by the bank's API in that call. Every tool call is visible live and retained. This is the direct engineering answer to Moffatt v Air Canada — see objections below.

Process A complete audit trail, per call

Each session leaves a masked transcript and a structured record: outcome, verification attempts, every tool call with masked arguments, escalation reference. The supervisor dashboard reads the same artefact QA and POPIA data-subject requests would.

Process Human escalation is a feature

An explicit escalation tool files a ticket and hands the consultant a summary — nobody repeats themselves. The EU AI Act's Article 50 duty (disclose it's an AI) is satisfied by a deterministic opening line — fixed text, not LLM-generated.

Contractual Residency & training exclusions

Production options range from self-hosted media to a fully self-hosted LLM — see below — for strict POPIA cross-border postures. Enterprise LLM terms exclude customer data from training — the DPA goes on the table in meeting one regardless of which posture is chosen.

Standard PCI DSS posture for card data

Card numbers and CVV must never enter recordings or transcripts. PCI SSC telephone-payment guidance endorses DTMF capture ("type it on your keypad") — supported by the telephony layer, so payment data bypasses the transcript entirely.

Deliberate omission We do not authenticate by voice

AI voice cloning defeated a major UK bank's voice-ID in a 2023 journalist demonstration, and the SARB/FSCA flag deepfake audio as a live consumer-protection risk. We treat the voice channel as untrusted input and verify with knowledge/possession factors instead. Saying this out loud earns credibility: the design assumes the adversary has AI too — and the agent never skips the check, however good the sob story.

For banks where no third-party API call is acceptable

Data sovereignty: self-host the model itself

Every guarantee so far assumes an LLM API call leaves the bank's network — even to an enterprise-contracted provider. LLM_PROVIDER=self_hosted removes that leg entirely: the model runs on the bank's own GPUs, in the bank's own data centre. It's a .env change in this codebase today, not a rewrite.

This is not hypothetical — two banks are already doing it. BNP Paribas renewed its Mistral AI partnership in May 2026 naming KYC specifically — "where data sensitivity, security and compliance requirements are particularly high" — for on-premises deployment within the Group's own data centres (BNP Paribas, primary). HSBC announced a multi-year Mistral AI partnership in Dec 2025 to run "self-hosted AI models that operate on HSBC's internal technology systems," with customer-facing work explicitly in scope (HSBC, primary). Lead with these two names before a single benchmark number.

Licenses to actively avoid for this pitch

Avoid Gemma 3

Google's Prohibited Use Policy restricts unlicensed "financial… professional practices" (similar to Llama's clause) — but the real problem is different: a flow-down obligation (you must contractually pass Google's restrictions to every downstream user of your product) plus Google's unilateral right to remotely restrict or terminate access. A real operational risk if a bank's call-centre uptime depends on it. Caveat: secondary sources reference a newer "Gemma 4" with Apache-licensed weights — unverified against Google's own docs at time of writing.

Avoid Command R+

Cohere's CC-BY-NC 4.0 license is non-commercial only — production needs a separate paid Cohere enterprise agreement, so it isn't "open" for this purpose as-is. Note: Cohere's newer Command A+ (May 2026) dropped this restriction and ships under Apache 2.0 — worth a second look.

Latency reality, said before the bank's engineers ask: a single 24GB GPU comfortably runs 7B–14B models in real time — fine for a pilot's call volume. But 14B+ models degrade sharply under concurrent load on one card (one benchmark: a 14B model's time-to-first-token went from 3.3s to 34s between 1 and 25 simultaneous requests on a single A100). Budget multi-GPU before promising production concurrency, and load-test on the bank's actual hardware before it goes in a contract.

The rest of the voice pipeline, fully on-prem

The LLM carries the most attack surface (it sees the full conversation and calls tools), so it's the priority — but STT/TTS can move on-prem too: faster-whisper (MIT) for speech-to-text, the standard self-hosted choice, and Kokoro-82M (Apache 2.0, ~28ms first-audio latency) for text-to-speech — the best license+speed combination found. Not yet wired into this POC (it uses LiveKit Inference by default); the same OpenAI-compatible-endpoint pattern generalises to these as a roadmap item once the bank prioritises full-stack residency.

South Africa first

The regulatory position, in three cards

Nothing here blocks a deployment today — but each card is a talking point the bank's compliance team will raise.

No AI-specific law blocks this

SA regulates AI in finance through existing technology-neutral frameworks (FAIS, POPIA, prudential standards). The SARB PA + FSCA joint report (24 Nov 2025) is guidance; a discussion paper is the stated next step. National policy direction: strengthen existing sectoral laws — no standalone AI Act.

POPIA is the operative law

It governs personal-data processing, automated decision-making, profiling, and cross-border transfers. Section 71 restricts significant decisions made solely by automation.

"The agent executes service requests — check a balance, block a card, open a dispute — and makes no credit or adverse decisions; anything judgement-shaped goes to a human." ← say this sentence verbatim.

Governance expectations → our artefacts

The regulators expect board oversight, model risk management, explainability, and disclosure. Map each to an artefact we already produce: the audit trail, the eval suite, the deterministic disclosure line, the DPIA-ready data flows. If the bank has EU exposure: customer-service AI is limited-risk; Article 50 disclosure is built in.

Raise them before they do

The four objections — and the answers that hold

"What if it makes something up?" — the Air Canada precedent

In Moffatt v Air Canada (2024), a tribunal held the airline liable for a bereavement-fare policy its chatbot invented. "The AI said it, not us" failed as a defence.

Own it: the bank is liable for what the agent says — which is exactly why answers are grounded in system calls, policy questions come from the bank's approved FAQ corpus, and everything else escalates.

Then show the activity panel proving where each answer came from. The objection becomes the demo.

"So we can cut the call centre?" — the CBA cautionary tale

Commonwealth Bank of Australia declared 45 call-centre roles redundant in 2025 citing its voice bot; call volumes rose instead, and CBA publicly reversed and apologised.

Position it as overflow absorption, not replacement: the agent takes growth and spikes so consultants keep the complex, revenue-relevant conversations. Headcount decisions come after the containment data, not before it.

This framing also defuses union and works-council friction before it starts.

"Can someone deepfake their way in?"

Not through the agent. Voice is treated as untrusted input; verification is knowledge/possession-based today and plugs into the bank's OTP or app-push in production.

The AI answering the phone doesn't weaken caller authentication — it standardises it: the agent never skips the check, and every attempt (and lockout) is logged as a security event.

"Does our data train someone's model?"

No — enterprise API terms plus a data processing agreement exclude customer data from training; in-country or VPC inference is available for strict residency postures.

This is a contracts conversation, not a research problem.

"We don't want a third-party API call in the loop at all"

Then don't have one. LLM_PROVIDER=self_hosted runs the model itself on the bank's own GPUs — see the data-sovereignty section. BNP Paribas and HSBC both already run Mistral models this way.

This isn't a future roadmap slide — it's a working config switch in the codebase today.

"What about isiZulu, Afrikaans, Setswana?"

Multilingual speech models make languages a configuration surface, but accuracy per language and accent must be proven, not assumed.

We position language rollout as evaluation-gated: each language ships when it passes the same behavioural test suite English passes today. Banks respect a measured answer more than a checkbox.

The 10-minute meeting

Pitch playbook

  1. The queue problem 1 min

    Open with their numbers: call volumes, hold times, cost per call. If you don't have them, ask — the answers become the calculator inputs.

  2. Live call, happy path 2 min

    Thabo: verify → balance → "has my salary come in?" — with the activity panel visible the whole time.

  3. The refusals 2 min

    Balance before verifying; a transfer request; three failed verifications → lockout. This is the security pitch, shown not told.

  4. Escalation 1 min

    Sipho: investment-advice refusal → human handoff with a summary the consultant reads back.

  5. Supervisor view 2 min

    "Here's the dashboard your ops team gets" — containment, handle time, and the audit trail of the call you just made together.

  6. Evidence & guarantees 1 min

    The deployments table and the security one-pager — leave this site with them.

  7. The ask 1 min

    A scoped pilot on one call type (card blocks is ideal: high volume, low judgement) with containment/CSAT success criteria read off the same dashboard.
